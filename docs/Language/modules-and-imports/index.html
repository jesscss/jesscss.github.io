Every preprocessor lets you split styles across files and pull them back together. The concept is the same everywhere — the spelling and the scoping rules are what differ.

Modules in Jess

Jess's module system is @-compose. It imports another stylesheet under a namespace, evaluates it in isolated scope, and lets you configure it per import. Compiler at-rules are dash-prefixed so they never clash with future CSS.

@-compose './theme' as theme;

.box {
  color: $theme.primary-color;
}
  • Namespace is inferred from the file name, or set with as name (use as * to compose without a namespace — discouraged, since it's harder to trace where values come from).
  • with { … } provides configuration for this compose only. It overrides the module's defaults without changing how another compose of that module begins.
  • set { … } establishes a one-time, compilation-wide configuration for the resolved module. Later composes of that same module reuse those values.
  • @-compose (reference) imports values/mixins without rendering rules (until extended); (protected) forbids downstream extension; (export) re-exports what it imports.
  • @-export './x'; forwards a module's API to consumers (Sass @forward).

See the full at-rule reference for every flag.

Scripts and data: @-use / @-from​

Use @-use (Sass-module style) or @-from (ESM style) to pull values from JavaScript, TypeScript, or JSON — and to load Jess's built-in #less/* and #sass/* modules.

@-use './my-module.js';                       // namespace inferred
@-from './my-module.js' import (myFunc);      // named import

.box {
  width: $my-module.width;
  value: $myFunc();
}

JSON is a pure data import; evaluating local/package JS/TS while compiling needs @jesscss/plugin-js (which isolates execution on Deno).

@-import is a legacy bridge (discouraged)​

Jess also supports @-import — the direct analog of Less/Sass @import — but it emits a deprecation warning. It re-exports everything from the imported file and changes the evaluation context on every call, which makes sources hard to reason about. Reach for @-compose instead; keep @-import only for mechanical migration.

@-import './my-file.jess'; // works, but warns — prefer @-compose

Plain CSS @import still passes through​

A native CSS @import — one that targets a URL or a .css file — is not a source include. It is emitted verbatim so the browser fetches it at runtime.

@import url("https://fonts.example/css?family=Inter");
@import "reset.css";

.jess has no import heuristic. A bare @import is always a plain CSS at-rule — @import "foo", @import "foo.less", and @import "foo.css" all pass straight through to the output. Folding a stylesheet in and evaluating it at compile time is done only by an explicit, dash-prefixed compiler at-rule:

  • @-compose — the module system (isolated scope, namespaced). Prefer this.
  • @-import — always folds and evaluates the file in place (the leaky legacy shape; warns — reach for @-compose).

The dash is required. Bare @use, @compose, @forward are not valid Jess at-rules — only @-use, @-compose, @-export are. That's the whole point of the dash: @import stays pure CSS, and anything that touches the compiler is unambiguously marked.
